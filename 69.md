是的，**现在可以开始扩另外的 regression case 了**。而且从现在开始，regression 只按一个标准做：

```text
真实 crate + pinned vulnerable version + 公开 advisory/issue + 真实 public API reproducer
```

不要再放手写模拟。手写的只能放 `micro/taxonomy`。

## 1. 下一批 regression 建议顺序

我建议按这个顺序扩：

| 优先级 | case                                     |       是否有官方 fixed | 为什么先做                                                             |
| --: | ---------------------------------------- | ----------------: | ----------------------------------------------------------------- |
|   1 | `toodee` / RUSTSEC-2021-0028             |       有，`>=0.3.0` | 有 vulnerable/fixed pair，最适合做论文 regression pair                    |
|   2 | `direct_ring_buffer` / RUSTSEC-2025-0105 |       有，`>=0.2.2` | Miri confirmed，适合补 oracle/RQ5                                     |
|   3 | `insert_many` / RUSTSEC-2021-0042        | 无 patched version | 和 `slice-deque` 一样是 panic-after-unsafe double drop，但没有 fixed pair |
|   4 | `through` / RUSTSEC-2021-0049            | 无 patched version | API 简单，`ptr::read + closure panic`，适合补一个 clean double-free case   |

`toodee` advisory 明确说 affected function 是 `toodee::TooDee::insert_row`，影响 `<0.3.0`，修复版本是 `>=0.3.0`；其问题是插入行时移动元素、复制 ownership，如果 iterator panic，可能 double free，或者暴露未初始化/已释放元素。([rustsec.org][1])
`direct_ring_buffer` advisory 明确说 affected function 是 `direct_ring_buffer::create_ring_buffer`，影响 `<=0.2.1`，修复版本是 `>=0.2.2`；问题是 `Vec::with_capacity` 后 `set_len` 造成未初始化内存，且 advisory 说该问题已用 Miri 确认。([rustsec.org][2])
`insert_many` advisory 说明 affected versions 使用 `ptr::copy` 移动 vector 元素腾位置，复制 ownership；如果 iterator 的 `.next()` panic，vector 会 drop 同一元素两次，并且没有 patched version。([rustsec.org][3])
`through` advisory 说明 `through/through_and` 对 mutable reference 调用 `ptr::read` 复制 ownership，然后调用 mapping function；如果 mapping function panic，原对象和复制对象都会被 drop，导致 double free，而且没有 patched version。([rustsec.org][4])

## 2. 先做哪个？

**先做 `toodee`。**

原因很简单：它有真实 vulnerable/fixed pair。你的 `slice-deque` 因为没有官方 patched version，只能做 vulnerable-only。接下来必须补一个 **有官方修复版本的 pair**，否则 regression suite 会被质疑“只测漏洞版，不测修复后是否消失”。

建议新增：

```text
benchmarks/regression/rsec_toodee_2021_0028_vuln
benchmarks/regression/rsec_toodee_2021_0028_fixed
```

依赖大致是：

```toml
# vuln
toodee = "=0.2.x"

# fixed
toodee = "=0.3.0"
```

注意：具体 vulnerable 版本你本地要用 `cargo search toodee` 或 crates.io 确认。RustSec 只告诉我们 `<0.3.0` 受影响、`>=0.3.0` 修复。不要随便写 `=0.2.0` 进论文，必须本地能构建并能触发。

## 3. `toodee` case 应该怎么设计

### vulnerable case

目标不是模拟 `TooDee`，而是直接调用真实 API：

```rust
use toodee::TooDee;

let mut td = TooDee::init( /* rows, cols, init value */ );
td.insert_row(index, PanicIterator::new(...));
```

核心 reproducer 要满足：

```text
1. 使用 TooDee::insert_row；
2. iterator 实现 ExactSizeIterator；
3. len() 返回会让 TooDee 预留/移动元素；
4. next() 在插入过程中 panic；
5. DropProbe 或 BoxProbe 观测 double drop / abort / sanitizer / replay evidence。
```

这个 case 的 expected 应该是：

```yaml
ground_truth:
  primary_label: PanicAfterUnsafe
  relation: AfterUnsafe
  oracle_verdict: Unknown
  harness_status: ValidHarness
  security_relevant: true
  oracle_confirmable: true
  expected_reached_count: 1

dangerous_categories:
  - AllocationOwnership
  - DropInvariant
```

和 `slice-deque` 一样，先不要急着写 `AddressSanitizerDoubleFree`。只有 ASan/Miri 真跑出来，才改 oracle verdict。

### fixed case

固定版本用 `toodee >=0.3.0`。fixed case 应该仍然调用真实：

```rust
TooDee::insert_row
```

同样传入会 panic 的 iterator。预期应该是：

```yaml
ground_truth:
  primary_label: ContractPanic
  relation: NoneObserved
  oracle_verdict: NoOracleFinding
  harness_status: ValidHarness
  security_relevant: false
  oracle_confirmable: true
  expected_reached_count: 0
```

如果 fixed 版本仍然 panic，这是正常的：predicate/iterator panic 是用户代码行为。关键是：

```text
fixed 版本不应出现 double drop / unsafe precondition violation / abort / oracle finding
```
