## 一、你的论文思路可以怎么表述

你的核心问题不是“Rust 会不会 panic”，而是：**一个 crate 中的 panic 行为，与危险操作路径之间到底是什么关系；哪些 panic 只是正常契约检查，哪些 panic 会阻断危险路径，哪些 panic 出现在危险操作之后，哪些案例已经被外部内存 oracle 证实为真实内存安全 bug。** 这条主线已经体现在你现在的数据模型里：系统同时建模了 `PanicRelation`、`CaseClass`、静态站点映射、运行时 trace，以及 oracle 证据。`PanicRelation` 里已经区分了 `NoDangerousSiteReached`、`BeforeUnsafe`、`AfterUnsafe`、`InsideUnsafeApprox`、`Unknown`，而 `CaseClass` 已经细化为 panic-only、unsafe reachable、oracle-confirmed 三大类。([GitHub][2])

从论文角度，你这套系统可以概括成一个 **Rust Dangerous Path Reachability** 框架：
第一步，静态识别 crate 中“可能危险”的站点和“可能 panic”的站点；
第二步，在 benchmark 或测试执行时记录“哪些危险站点被 hit，是否发生 panic，以及 panic 的位置/消息”；
第三步，结合 ASan/Miri 之类的 oracle，把“可疑路径”进一步升级为“已证实 bug”；
第四步，把所有证据融合成 taxonomy，输出一个对研究者友好的结论。这个思路已经不是单纯的 fuzzing，也不是单纯的静态分析，而是**静态站点发现 + 轻量运行时追踪 + 外部 oracle 证据融合**。([GitHub][3])

## 二、你的系统设计流程

### 1. 元数据收集与站点发现

`rustdpr-analyzer` 已经把“先收集 crate 元数据，再构建 site map”这条流程做出来了。`collect_metadata(crate_dir)` 通过 `cargo metadata` 提取 crate 名称、版本、targets 等信息；`build_site_map(crate_dir, crate_name)` 则递归遍历 `.rs` 文件，建立危险站点和 panic 站点映射。([GitHub][4])

从 `site_locator.rs` 现在的访问逻辑看，危险站点至少已经覆盖了这些类别：
`unsafe fn`、`unsafe block`、`extern`/FFI 声明、`transmute` 调用，以及索引表达式候选；panic 站点至少覆盖了 `panic!`、`assert!/debug_assert!`、`todo!`、`unimplemented!`、`unwrap`、`expect`、索引表达式运行时检查。也就是说，你现在的静态层已经能把“危险操作候选”和“panic 候选”同时打到同一张 `SiteMap` 里。([GitHub][3])

### 2. 运行时追踪

`rustdpr-trace` 已经实现了最小但完整的运行时事件记录机制：`init_trace(path)` 初始化 trace 文件，`hit(site_id)` 记录命中的危险站点，`install_panic_hook()` 则通过 panic hook 把 panic 的 message、file、line 记成 `TraceEvent::Panic`。这使得你的 trace 不只是“是否崩溃”，而是包含 **危险路径到达信息 + panic 发生信息** 的双轨事件流。([GitHub][5])

### 3. 分类器与 taxonomy

`rustdpr-classifier` 现在已经不只是早期的粗分类器了。它会读取 trace 中的 `Hit` 和 `Panic` 事件，抽取 `reached_site_ids`、`panic_message/file/line`，然后基于 `site_map` 与 `oracle` 做融合分类。更重要的是，`ClassificationResult` 已经补齐了 `oracle_confirmed`、`oracle_results`、`reached_dangerous_site`、`panic_observed`、`taxonomy_reason` 这些字段，这让分类结果已经具备论文实验和报告展示所需的解释性。([GitHub][2])

你的 taxonomy 现在已经成形：
panic-only 包括 `NormalContractPanic`、`BlockingPanic`、`HarnessMisuse`；
unsafe reachable 但未证实的包括 `ReachableUnsafeNoPanic`、`PanicAfterUnsafe`、`SuspiciousCandidate`；
oracle-confirmed 细化到 `OracleConfirmedDoubleFree`、`OracleConfirmedUseAfterFree`、`OracleConfirmedOutOfBounds`、`OracleConfirmedMemoryCorruption`、`OracleConfirmedUndefinedBehavior`，并保留 `OracleConfirmedMemoryBug` 作为兜底。这个层级化 taxonomy 已经非常接近论文里“结果标签体系”的形态。([GitHub][2])

### 4. Oracle 证据融合

你已经单独做了 `rustdpr-oracle` crate，用于把外部工具输出解析为统一的 `OracleResult`。`OracleFinding` 当前已经带有 `oracle`、`verdict`、`message`、`stack`、`location`、`raw_message`，说明你已经不只是想要一个布尔值，而是在保留可追溯证据。ASan parser 已经会抓取摘要行、stack frame 和 primary location；Miri parser 也会产出同样结构的字段。([GitHub][2])

分类器里也已经有了“强 oracle override”的逻辑：`strongest_oracle_case` 会按 `DoubleFree → UseAfterFree → OutOfBounds → MemoryCorruption → UndefinedBehavior → InvalidFree` 的优先级，选出最强的 oracle verdict，再映射成更细的 `CaseClass`。这意味着你的系统现在已经具备“启发式结果可被外部证据升级”的能力。([GitHub][6])

### 5. CLI 与报告生成

`rustdpr-cli` 已经把整条主链串起来了：`collect`、`analyze-sites`、`classify --oracle`、`report --oracle`、`oracle-parse`，以及第十七阶段加入的 `summarize-class`。这说明系统已经从“库功能”走到了“可执行 pipeline”。([GitHub][7])

`rustdpr-report` 也已经接收 `ClassificationResult + OracleResult`，并在报告中展示 dangerous sites、panic sites、trace events、classification、`Panic Observed`、`Reached Dangerous Site`、`Taxonomy Reason`、`Taxonomy Summary` 和 `Oracle Findings`。这意味着你的输出已经不只是供程序消费的 JSON，而是开始有研究报告属性。([GitHub][8])

## 三、目前已经实现的内容

从“工程完成度”看，你已经实现了一个可跑通的最小研究原型，而不是零散脚本集合。

第一，你已经完成了 **workspace 拆分与职责分层**。核心模型、分析器、追踪器、分类器、oracle parser、report renderer、CLI 都已经独立成 crate，这种结构很适合论文中描述“系统架构”。([GitHub][9])

第二，你已经完成了 **静态站点发现** 的第一版。它能够识别 unsafe/FFI/transmute/index 等危险站点，以及 panic/assert/unwrap/expect 等 panic 站点，并产出 site map。虽然还不是非常精细，但已经足以支撑危险路径 reachability 的最小实验。([GitHub][3])

第三，你已经完成了 **运行时证据采集**。trace 现在能够持久化 hit 和 panic 事件，这为“panic before/after unsafe”关系分类提供了运行时依据。([GitHub][5])

第四，你已经完成了 **taxonomy 细化后的分类模型**。`CaseClass` 与 `ClassificationResult` 现在已经不是临时枚举，而是能支撑后续 benchmark 断言、报告展示、以及论文实验表格的结构化结果。([GitHub][2])

第五，你已经完成了 **oracle 接入**。ASan/Miri 都有解析器，CLI 也支持 `oracle-parse`，分类器还能利用 oracle 做 override。这个点非常重要，因为它让你的研究从“可疑路径分析”升级成“可疑路径 + 已证实 bug”的二级证据体系。([GitHub][10])

第六，你已经完成了 **Windows 友好的自动化执行链**。`run_asan.py` 用 `RUSTFLAGS=-Zsanitizer=address` 和 `ASAN_OPTIONS` 调用 nightly cargo test，`run_miri.py` 用 `cargo +nightly miri test` 收集日志，`run_oracle_case.py` 则把 analyze-sites → run_asan → oracle-parse → classify → validate → report 串成一条完整流程。([GitHub][11])

第七，你已经开始把 **benchmark 与 taxonomy 预期结果** 对齐。`mb_double_free`、`mb_heap_uaf`、`mb_oob_raw` 的 `expected.yaml` 已经不再只写一个笼统类名，而是明确要求 `class`、`oracle_confirmed`、`panic_observed`、`reached_dangerous_site` 和 `oracle_verdicts`。这已经很像论文实验里的 ground truth 断言格式了。([GitHub][12])

第八，你已经开始加入 **taxonomy benchmark**。workspace 里已经把 `benchmarks/taxonomy/panic_before_unsafe` 纳入成员，而且这个 benchmark 代码里确实包含一个 panic 前置分支，以及后续的真实 `unsafe` 解引用和 `hit("S0001")`，这正对应你前面想验证的 `BlockingPanic` 场景。([GitHub][9])

## 四、按论文流程看，你现在处在什么阶段

如果按“论文研发流程”来分阶段，你现在已经完成了：

* **问题定义**：panic 与危险路径关系不是单一标签，而是分层 taxonomy。([GitHub][2])
* **核心数据模型**：site map、trace、classification、oracle result 已统一。([GitHub][2])
* **系统原型**：静态分析、运行时追踪、oracle 解析、分类、报告、脚本自动化均已形成闭环。([GitHub][3])
* **初步实验资产**：oracle benchmarks 和 taxonomy benchmark 已开始覆盖关键类别。([GitHub][9])

也就是说，你现在已经从“设计和搭骨架”跨到了“可以系统做实验”的阶段。

## 五、目前还缺什么

这里我分成“工程缺口”和“论文缺口”。

### 工程缺口

第一，**静态危险站点识别还偏粗**。从当前 `site_locator.rs` 看，你已覆盖 unsafe fn、unsafe block、FFI、transmute、index candidate，但还没有明显看到更细的 Rust 内存危险模式识别，比如原始指针解引用、`from_raw`/`into_raw` 配对、`get_unchecked`、裸指针 arithmetic、`MaybeUninit`、`ManuallyDrop`、自定义 allocator 相关模式。现在的 analyzer 已经够做原型，但离“更有说服力的论文实验”还差一层更贴近真实 unsafe bug 习惯用法的规则。([GitHub][3])

第二，**relation 细粒度还没有真正用满**。`PanicRelation` 已经定义了 `BeforeUnsafe` 和 `InsideUnsafeApprox`，但从 classifier 当前逻辑片段看，主要仍围绕 `NoDangerousSiteReached`、`AfterUnsafe` 和 `Unknown` 在做分类。也就是说，taxonomy 枚举已经走在逻辑实现前面，接下来应该让 classifier 真正利用时间顺序或站点邻近关系把 `BeforeUnsafe`、`InsideUnsafeApprox` 判得更可靠。([GitHub][2])

第三，**oracle pipeline 目前主要走 ASan**。`run_oracle_case.py` 当前流程只显式调用了 `run_asan.py`，再把 `asan.log` 送去 `oracle-parse`；`run_miri.py` 存在，但没有并入这条默认主流程。也就是说，你已经有了双 oracle 能力，但默认实验链还没有真正做到 ASan+Miri 融合。([GitHub][13])

第四，**runner 的 validation 还没有校验 `oracle_verdicts`**。现在 `validate_classification()` 只校验 `class`、`oracle_confirmed`、`panic_observed`、`reached_dangerous_site`，但并没有把 `expected.yaml` 中的 `oracle_verdicts` 和实际 `oracle_results` 对上。既然你已经把 expected 写细了，这一步最好也补上。([GitHub][14])

第五，**缺少聚合统计层**。你现在能跑单 case 并输出单份 markdown report，但还没有看到“批量跑所有 benchmark 后生成一张总表/CSV/summary.md”的层。对论文来说，这一层非常关键，因为最终要展示的不是单个案例报告，而是整体 taxonomy 分布、oracle-confirmed 占比、误报/未证实比例等。这个部分我在当前仓库里还没看到。([GitHub][14])

### 论文缺口

第一，**还缺评价指标的定义**。当前代码已经能产出分类，但论文里还需要明确：你要评估的是 taxonomy 准确率、oracle-confirmed 发现率、panic-blocking 识别率，还是“可疑路径到真实 bug 的转化率”。这些指标目前还没有在仓库层面被固化。

第二，**还缺基线和对照**。论文如果只展示“我的系统能分类”说服力不够。你需要至少一种 baseline，例如：

* 只靠 panic 是否发生的粗分类；
* 只靠静态 dangerous site 的 reachability；
* 不带 oracle 的版本 vs 带 oracle 的版本。

你现在的架构已经很适合做这个 ablation，但还没有看到对应脚本或实验表。

第三，**还缺 benchmark 规模化**。现在已有 micro、oracle、taxonomy 三类 benchmark，但数量还不够多。要支撑论文，最好每个 taxonomy 类别至少有若干个稳定样例，并覆盖不同危险模式。workspace 已经为这件事打好了结构基础。([GitHub][9])

第四，**还缺文字化设计文档**。我没在仓库里成功拿到 README，顶层页面也显示没有 description。对论文写作和答辩来说，你现在非常需要一份仓库内的 `README.md` 或 `docs/design.md`，把问题定义、架构图、数据流、跑法、taxonomy 说明写出来。仓库首页目前没有这些用户可见说明。([GitHub][1])

## 六、我建议你接下来怎么做

最优先的不是再加新 crate，而是把现有原型打磨成“可实验、可复现、可写论文”的版本。

第一步，补 **relation 细化实现**。把 `BeforeUnsafe` 和 `InsideUnsafeApprox` 真正用起来，让 `BlockingPanic` 和 `PanicAfterUnsafe` 的边界更清楚。现在你的 taxonomy 很好，但 relation 逻辑还可以再更精确。([GitHub][2])

第二步，把 **Miri 并入默认 oracle 流程**。当前 runner 默认只走 ASan，你可以让 `run_oracle_case.py` 同时调用 `run_miri.py`，然后 `oracle-parse` 支持同时读 `--asan-log` 和 `--miri-log`，最后合并 findings。这样论文里就能说你不仅依赖单一 sanitizer。([GitHub][13])

第三步，补 **validation 对 oracle_verdicts 的校验**。你已经在 expected 里写了更细的 ground truth，再把 runner 补完，这样 benchmark 就变成真正可回归验证的实验资产。([GitHub][14])

第四步，做 **批量运行与统计汇总**。新增一个总 runner，扫描 `benchmarks/oracle` 和 `benchmarks/taxonomy`，统一产出：

* 总 case 数
* 每个 `CaseClass` 的数量
* oracle-confirmed 数量
* 未证实的 `SuspiciousCandidate` 数量
* 失败 case 列表
  这是论文实验章节最需要的东西。

第五步，扩充 **taxonomy benchmark 集**。现在至少已经有 `panic_before_unsafe` 一个 taxonomy case；接下来你还应补：

* `normal_contract_panic`
* `panic_after_unsafe`
* `reachable_unsafe_no_panic`
* `harness_misuse`
  这样你的 taxonomy 才能完整覆盖。([GitHub][9])

第六步，补 **仓库文档与论文图示**。把系统架构图画出来：
`Analyzer -> SiteMap`
`Trace runtime -> TraceLog`
`ASan/Miri -> OracleResult`
`Classifier -> ClassificationResult`
`Report -> Markdown/summary`
这张图几乎可以直接放进论文方法章节。

## 七、你这篇论文可以怎么写

你现在的代码已经足够支撑一篇结构清晰的论文，主线可以写成：

**题目方向**
Rust 中 panic 与危险路径关系的分类分析：结合静态站点发现、运行时追踪与 oracle 证据融合的方法

**论文结构**

1. 背景与问题：Rust 并不等于没有内存风险；panic 与 unsafe 路径关系复杂。
2. 问题定义：给定 crate/test execution，判定 panic 与危险路径关系，并区分正常 panic、阻断 panic、危险后 panic、可疑未证实、已证实 bug。
3. 方法：静态站点发现、trace 记录、oracle 解析、taxonomy 分类。
4. 实现：workspace 架构、CLI、Python runner、report。
5. 实验：micro + oracle + taxonomy benchmarks；展示 taxonomy 分布和 oracle 证实能力。
6. 讨论：启发式误差、relation 近似、oracle 覆盖限制、对真实 crate 的可迁移性。

这条写法和你现在仓库的实现状态是吻合的，不需要再强行换题。
